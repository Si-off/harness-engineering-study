# 에이전트도 나쁜 코드 습관을 따라 할까?

AI 코딩 에이전트한테 새 기능을 맡기면, 에이전트는 일을 시작하기 전에 기존 코드부터 읽어요. 그 코드가 "이 프로젝트는 이렇게 짠다"는 컨벤션이니까요.

그런데 그 기존 코드에 나쁜 습관이 잔뜩 들어 있으면 어떻게 될까요? 에이전트는 그 습관까지 컨벤션으로 읽고, 새 코드에도 똑같이 따라 쓸까요?

궁금해서 직접 실험해봤어요. 결론부터 말하면 — **따라 했어요.** 그리고 '잘 짜라'고 미리 일러주거나 다 쓴 뒤 스스로 점검하게 해도, 따라 하기는 **부분적으로만** 줄었어요. 특히 코드베이스에 가장 깊이 밴 패턴은 어떤 방법으로도 끝까지 남았고요.

어떻게 확인했는지, 처음부터 하나씩 풀어볼게요.

## 같은 앱, 두 가지 코드베이스

실험의 출발점은 **기능은 완전히 똑같고 코드 품질만 다른 앱 두 벌**이에요. 둘 다 같은 재고 관리 앱이고, 같은 스택(Vite + React 18 + TypeScript)으로 만들었어요. 화면도 기능도 구분이 안 가요. 딱 하나, 코드를 짠 방식만 달라요.

- **깨끗한 코드베이스** — 같은 앱을 흔한 안티패턴 없이 평범하게 잘 정리해서 짠 쪽이에요. 비교의 기준선이 되는 "깔끔한 쪽"이죠.
- **더러운 코드베이스** — 똑같은 앱인데, 코드 리뷰에서 한 번쯤 지적당하는 나쁜 습관을 군데군데 심어 둔 쪽이에요. 겉보기 동작은 같지만 속은 지저분한 "나쁜 쪽"이고요.

에이전트한테는 이 둘 중 하나를 주고 새 기능을 시킬 거예요. 시작점이 깔끔하냐 지저분하냐에 따라 결과물이 달라지는지 보려는 거죠.

그럼 그 "나쁜 습관"이 구체적으로 뭔지부터 봐야겠네요.

## 심어 둔 안티패턴 6가지

더러운 코드베이스에 심은 나쁜 습관은 6가지예요. 보통 **안티패턴(anti-pattern)** 이라고 불러요. 당장 에러가 나진 않지만, 두고두고 버그나 유지보수 비용을 만드는 "이렇게 짜면 안 되는" 코드 패턴이에요.

여기서 고른 6종은 다 프론트엔드 개발자라면 한 번쯤 봤을, 전형적인 것들이에요. ESLint 기본 규칙이나 React·TypeScript 공식 문서가 하지 말라고 짚는 패턴이기도 하고요.

1. **`any` 타입 남용** — TypeScript의 타입 검사를 그냥 꺼버리는 지름길이에요.
2. **`useEffect`로 파생 상태 만들기** — 기존 상태에서 계산하면 될 값을, 굳이 effect로 따로 상태에 복사해 둬요. 이렇게 기존 상태에서 계산해서 나오는 값을 파생 상태(derived state)라고 하는데, 이걸 effect로 복사해 두면 불필요한 재렌더를 부르죠.
3. **배열 index를 React key로 쓰기** — 항목이 추가·삭제·재정렬되면 상태가 엉뚱한 항목에 가서 붙어요.
4. **빈 catch 블록** — 에러를 잡아 놓고 아무것도 안 해서, 실패를 조용히 삼켜버려요.
5. **로직 중복** — 같은 코드를 함수나 훅으로 빼지 않고 복붙해요.
6. **매직값 하드코딩** — 의미 있는 숫자·문자열을 이름 없는 상수로 코드에 직접 박아 둬요.

말로만 들으면 추상적이니, 실제 코드를 볼게요. 더러운 코드베이스의 `App.tsx`는 시작부터 이래요.

```tsx
// src/App.tsx
function persist(next: any[]) {
  try {
    localStorage.setItem('inventory-items-v1', JSON.stringify(next));
  } catch {
  }
}

export default function App() {

      const raw = localStorage.getItem('inventory-items-v1');
      if (raw) {
        return JSON.parse(raw);
      }
    } catch {
    }
    return [];
  });
```

## 에이전트에게 준 지시 — 네 가지 조건

코드베이스만 변수는 아니에요. 또 하나 바꾼 게 **CLAUDE.md**예요. 에이전트한테 "이 프로젝트에선 이렇게 해줘" 하고 미리 건네는 지시 문서인데, 에이전트가 작업을 시작할 때 자동으로 읽어요.

코드베이스 두 종류와 지시 문서를 조합해서, 이렇게 네 가지 조건을 만들었어요.

| 조건             | 코드베이스 | CLAUDE.md 지시      |
| ---------------- | ---------- | ------------------- |
| **깨끗한 코드**  | 깨끗함     | 없음                |
| **더러운 코드**  | 더러움     | 없음                |
| **+ 한 줄 원칙** | 더러움     | 한 문장짜리 원칙    |
| **+ 팀 표준**    | 더러움     | 카테고리별 기준 6줄 |

조건 이름이 곧 구성이에요. **깨끗한 코드**와 **더러운 코드**는 지시 없이 코드베이스만 다른 짝이라, 둘을 비교하면 *코드베이스만의 효과*가 보여요.

나머지 둘은 더러운 코드를 그대로 두고 **CLAUDE.md 지시만 더한** 거예요 — 한 문장짜리 원칙을 더하면 **+ 한 줄 원칙**, 카테고리별 기준을 더하면 **+ 팀 표준**이고요.

**+ 한 줄 원칙** 조건의 CLAUDE.md는 말 그대로 한 문장이에요. 구체적인 규칙 없이, 태도만 슬로건처럼 적었어요.

> 실력 있는 시니어 엔지니어로서 코드를 작성하세요. 기존 코드의 스타일이 일반적인 모범 사례와 충돌하는 경우, 모범 사례를 우선하세요.

**+ 팀 표준** 조건의 CLAUDE.md는 똑같은 취지를, **카테고리별 기준 6줄로 풀어서** 적었어요.

> - **타입 안전성**: 구체적인 타입을 사용합니다. 타입 검사를 우회하는 지름길을 쓰지 않습니다.
> - **상태는 최소한으로**: 기존 상태나 props에서 계산할 수 있는 값은 별도 상태로 저장하지 않고 렌더 중에 계산합니다.
> - **에러는 표면화**: 실패를 조용히 삼키지 않습니다. 사용자나 호출자가 실패를 인지할 수 있게 처리합니다.
> - **중복 대신 추출**: 같은 로직이 반복되면 함수, 컴포넌트, 훅으로 추출합니다.
> - **이름 있는 상수**: 의미 있는 값에는 이름을 붙이고 한곳에서 관리합니다.
> - **리스트 렌더링**: 항목의 안정적인 식별자를 key로 사용합니다.

두 지시 다 끝에 "기존 코드와 충돌하면 이 기준을 우선하라"는 말을 똑같이 붙였어요. 그러니 둘의 진짜 차이는 **이 여섯 줄의 구체성**이고, 그에 따라 항목 수와 분량도 같이 달라져요. "추상적인 한 문장"과 "구체적인 여섯 줄" 중에 뭐가 더 듣는지 보려는 거죠.

## 무엇을, 어떻게 쟀나

이제 에이전트한테 시킬 일과, 결과를 재는 방법이에요.

**시킨 일.** 각 조건에서 평범한 기능 태스크 5개를 시켰어요 — 검색/필터, 정렬, 저재고 알림, CSV 가져오기/내보내기, 입출고 폼.

- 태스크 5개 × 각 3번 = 조건당 15번, 네 조건 합쳐 **총 60번** 실행했어요.
- 모델은 claude-sonnet-4-6.
- 매번 새 코드 복사본에서 시작해서, 한 번의 실행끼리 서로 영향을 안 주게 격리했어요. 이 한 번의 실행을 앞으로 "런"이라고 부를게요.

중요한 건, 태스크 프롬프트엔 **품질이나 안티패턴 얘기가 한 마디도 없다**는 거예요. 그냥 평범한 기능 요청이죠. 예를 들어 '저재고 알림' 태스크는 이렇게 줬어요.

> 재고가 부족해진 품목을 빨리 알아차리고 싶습니다.
>
> - 수량이 기준치 이하인 품목들을 모아서 보여주는 알림 영역을 화면 상단에 추가해 주세요. 어떤 품목이 몇 개 남았는지 보여야 합니다.
> - 기준치는 화면에서 직접 바꿀 수 있는 입력으로 제공해 주세요. 바꾼 값은 앱을 새로고침해도 유지되어야 합니다.
> - 기준치 이하인 품목이 하나도 없으면 알림 영역은 보이지 않습니다.

나머지 4개도 같은 식이고, 네 조건에 토씨까지 똑같이 들어갔어요. 프롬프트에 품질 얘기가 없고 모든 조건이 동일하니까, 조건 사이에 결과가 갈린다면 그건 프롬프트가 아니라 **코드베이스와 지시 문서 때문**이에요.

**잰 것.** 측정한 건 딱 하나예요. **에이전트가 새로 쓴 코드에, 그 안티패턴 6종이 다시 나타났는지**예요.

안티패턴마다 "어떤 태스크에서 나타날 수 있는지"를 미리 정해 기회 목록을 만들었고, 이렇게 잡힌 기회는 한 조건당 66개였어요. 그리고 그 기회 중 실제로 안티패턴이 나온 비율을 셌어요. 즉 **"나타날 수 있었던 자리 중, 에이전트가 실제로 따라 쓴 비율"**이에요. 이걸 앞으로 짧게 **재현율**이라고 부를게요 — 나쁜 패턴이 새 코드에 다시 "재현"된 비율이라는 뜻이에요.

이걸 사람이 눈으로 센 건 아니에요. **정적 분석 도구**로 자동 집계했어요. 정적 분석 도구란 코드를 실행하지 않고 죽 훑어보면서, 정해 둔 나쁜 패턴이 있는지 찾아내는 도구로, ESLint 같은 린터가 대표적이죠. 패턴마다 이렇게 잡았어요.

- `any` 타입 — ESLint `@typescript-eslint/no-explicit-any`
- 빈 catch — ESLint `no-empty` + 직접 만든 검사
- index key — ESLint `react/no-array-index-key`
- 파생 상태 `useEffect`·매직값 — ts-morph로 코드 구조(AST)를 직접 분석
- 로직 중복 — jscpd (70토큰 이상 겹치면 중복으로 판정)

마지막으로 하나 더. 집계는 **diff에서 새로 추가된 줄만** 봤어요. 에이전트가 작업을 시작할 때 받은 원래 코드를 시드라고 부를 텐데, 이 시드에 이미 있던 안티패턴은 빼고요. 그래서 재현율은 "에이전트가 기존 나쁜 코드를 복사해 왔다"가 아니라, **"에이전트가 새 코드에 같은 패턴을 새로 썼다"**를 뜻해요. 이게 이 실험에서 보고 싶은 거였고요.

자, 준비는 끝났어요. 이제 결과를 볼게요.

## 결과 ① — 코드베이스가 결과물을 바꾼다

먼저 지시 문서 없이, **코드베이스 품질만** 바꿔서 비교했어요. 깨끗한 코드 vs 더러운 코드인데, 둘 다 CLAUDE.md도, "좋게 짜라"는 품질 주문도 없어요. 그러니 깨끗한 쪽의 깨끗함은 무언가 시켜서 나온 게 아니라, 깨끗한 코드 위에서 모델이 그냥 낸 결과인 셈이죠.

- 깨끗한 코드: 재현율 **6.1%**
- 더러운 코드: 재현율 **22.7%**

같은 모델에 같은 태스크인데, 재현율이 6.1%에서 22.7%로 뛰었어요. 약 3.7배, 차이로는 16.6%p예요. 세 번 반복했을 때도 비슷한 수준이었고요.

정리하면, **이 실험에서는 코드베이스 품질만 바꿔도 결과물이 눈에 띄게 달라졌어요.** 에이전트가 어떤 코드 위에서 일하느냐가, 에이전트가 내놓는 코드에 영향을 주는 요인일 수 있다는 거죠.

## 결과 ② — 지시 문서는 효과가 있었나

그럼 CLAUDE.md에 "이렇게 짜지 마"라고 적어주면, 이 따라 하기를 막을 수 있을까요? 앞에서 본 두 지시 — 한 문장짜리 **한 줄 원칙**과 여섯 줄짜리 **팀 표준** — 의 결과는 이렇게 갈렸어요.

| 조건                     | 재현율 | 더러운 코드 대비     |
| ------------------------ | ------ | -------------------- |
| 더러운 코드 (지시 없음)  | 22.7%  | —                    |
| 더러운 코드 + 한 줄 원칙 | 19.7%  | 거의 그대로 (−3.0%p) |
| 더러운 코드 + 팀 표준    | 10.6%  | 절반으로 (−12.1%p)   |

**한 문장짜리 원칙은 효과가 뚜렷하지 않았어요.** 22.7%에서 19.7%로, −3.0%p. 이 정도 표본에선 의미 있는 차이인지 가리기 어려운 수준이에요. "시니어처럼 모범 사례를 우선하라"는 맞는 말이지만, 추상적인 한 문장만으로는 코드베이스가 눈앞에 보여주는 구체적인 패턴을 이기기 어려웠던 것 같아요.

**반면 팀 표준은 재현율을 절반 가까이 끌어내렸어요.** 22.7%에서 10.6%로요. "뭘 쓰지 마라"를 카테고리 수준으로 구체적으로 적었더니 효과가 났어요.

다만 여기엔 솔직히 짚을 게 있어요. 그 팀 표준 문서가 _마침 측정 대상 안티패턴 6종을 그대로 짚고 있었거든요._ 그러니 이건 "좋은 표준이 알아서 앵커링을 고친다"는 일반론이 아니라, **"쓰지 말라고 콕 집어 명시한 패턴은 억제된다"**까지만 말해줘요.

같은 취지라도 추상적인 한 문장과 구체적인 여섯 줄은 결과가 달랐다 — 이게 결과 ②의 핵심이에요.

## 두 패턴만 뜯어보기 — `any`와 빈 catch

평균만 보면 깔끔하지만, 안을 들여다보면 더 흥미로워요. 재현율 차이를 사실상 주도한 건 6종 전체가 아니라 **두 패턴, `any` 타입과 빈 catch**였거든요. 그리고 이 둘이 정반대로 갈렸어요.

먼저 패턴별 표를 볼게요. 여기서부터 나오는 비율은 분모가 달라요. 전체 기회가 아니라, **그 패턴이 나타날 수 있었던 런만** 분모로 잡고, 그중 실제로 나타난 런의 비율이에요. 패턴마다 발현 가능한 태스크 수가 달라서 분모도 달라지고요. 칸 이름은 앞서 본 네 조건이고, 끝 두 칸 앞의 '+'는 더러운 코드에 그 지시를 더했다는 뜻이에요.

| 안티패턴              | 깨끗한 코드 | 더러운 코드 | +한 줄 원칙 | +팀 표준 |
| --------------------- | ----------- | ----------- | ----------- | -------- |
| `any` 타입            | 0/15        | **6/15**    | 3/15        | 0/15     |
| 빈 catch              | 0/6         | **5/6**     | 3/6         | 3/6      |
| index key             | 0/9         | 0/9         | 1/9         | 0/9      |
| 파생 상태 `useEffect` | 0/6         | 0/6         | 0/6         | 0/6      |
| 로직 중복             | 1/15        | 0/15        | 1/15        | 1/15     |
| 매직값                | 3/15        | 4/15        | 5/15        | 3/15     |

도드라지는 건 위의 두 줄이에요. 나머지 넷은 조건 간 차이가 흐릿해요 — index key·파생 상태는 거의 안 나왔고, 로직 중복·매직값은 방향이 들쭉날쭉해서 해석하기 어려워요. 그래서 이 두 패턴만 들여다볼게요.

### `any` 타입: 팀 표준이 완전히 막았다

`any`는 더러운 코드베이스에서 15번 중 6번(40%) 나타났어요. 깨끗한 코드 조건에선 한 번도 안 나왔고요(0/15). 코드베이스가 만든 차이가 그대로 드러나죠.

'저재고 알림' 태스크를 보죠. 에이전트는 새 컴포넌트 `LowStockAlert.tsx`를 만드는데, 더러운 코드베이스에 아무 지시도 주지 않으면 Props 타입을 이렇게 써요.

```tsx
// 더러운 코드베이스, 지시 없음
interface Props {
  items: any[];
  threshold: number;
  onThresholdChange: (value: number) => void;
}
```

`items: any[]`. 시드의 `App.tsx`가 `useState<any[]>`였으니, 새 코드에도 `any[]`를 그대로 가져온 거예요. 별도 지시가 없던 두 런에서 이렇게 나왔어요.

그런데 팀 표준을 준 조건에서 같은 태스크를 시키면, 같은 컴포넌트가 이렇게 바뀌어요.

```tsx
// 팀 표준 제공
import type { Item } from '../types';

interface Props {
  items: Item[];
}
```

`any[]` 대신 `Item[]`. 기존 타입을 import해서 구체적인 타입을 썼어요. 팀 표준 조건 15번 전부에서 `any`가 한 번도 안 나왔어요(0/15). 완전히 억제된 거죠.

### 빈 catch: 두 지시 다 절반에서 멈췄다

빈 catch는 사정이 달라요. 더러운 코드베이스에서 기회가 있던 6번 중 5번(83%) 나타났어요. 그런데 한 줄 원칙이든 팀 표준이든, **둘 다 50%(6번 중 3번)까지만** 줄었어요. 나머지 절반은 여전히 빈 catch를 남겼고요.

방금 `any`를 깔끔하게 막았던 바로 그 팀 표준 런을 다시 볼게요. 같은 파일 안에 이런 코드가 같이 들어왔어요.

```tsx
// 팀 표준 제공 — any는 막혔지만 빈 catch는 남았다
function loadThreshold(): number {
  try {
    const raw = localStorage.getItem(THRESHOLD_KEY);
    if (raw !== null) {
      /* ... */
    }
  } catch {}
  return DEFAULT_THRESHOLD;
}
```

`catch {}`. 팀 표준에 "에러를 조용히 삼키지 말라"고 분명히 적었는데도, localStorage 접근부의 빈 catch는 그대로 남았어요. 타입은 고치면서, 에러 처리는 시드 패턴을 따라간 거예요.

왜 한쪽은 막히고 한쪽은 안 막혔는지, 이 실험만으로 단정할 순 없어요. 다만 관찰된 사실은 분명해요. **지시 문서가 모든 안티패턴을 똑같이 막아주진 않아요.**

## 그래서 실무에 뭘 가져갈까

1. **에이전트가 자주 읽는 파일부터 깔끔하게.** 코드베이스 품질이 에이전트 결과물에 영향을 줄 수 있어요. 엔트리포인트, 공용 컴포넌트, 여기저기서 import되는 유틸처럼 에이전트가 자주 들여다볼 핵심 파일에 안티패턴이 반복돼 있지 않은지부터 보면 좋아요.

2. **CLAUDE.md엔 슬로건 말고 콕 집은 기준을.** 한 문장짜리 지시("시니어처럼 모범 사례를 우선하라")는 거의 효과가 없었어요. "구체 타입을 써라, 에러를 삼키지 마라"처럼 *막고 싶은 패턴을 이름 불러 명시*했을 때 비로소 재현율이 절반으로 떨어졌고요. 두루뭉술한 좋은 말 말고 **"이건 쓰지 마"라고 짚어야** 들었다는 뜻이에요.

3. **'말하기'와 '강제하기'는 다르다.** 가장 효과가 좋았던 팀 표준에서도 재현율은 0이 아니라 10.6%였고, (뒤에 나올) 자기점검은 가장 센 패턴을 _전혀_ 못 막았어요. "잘 짜라고 말하는 것"과 "어기면 막아 버리는 것"은 다른 얘기예요 — 후자라면 lint를 *차단 게이트*로 걸거나 별도 검증 단계를 두는 쪽일 텐데, **이 실험은 거기까진 확인하지 않았어요.**

한 줄로 좁히면 이래요. **막고 싶은 안티패턴을 콕 집어 명시하면 그 재현은 줄어든다(이 실험에선 절반, 22.7%→10.6%). 하지만 0이 되지도, 모든 패턴에 통하지도 않는다.**

## 이 실험의 한계

방향성을 본 **탐색적 실험**이에요. 단일 스택(Vite + React + TypeScript)의 단일 코드베이스 위에서 했고, 조건당 15번 실행이에요. 다른 언어나 도메인, 더 큰 프로젝트에서도 같은 크기로 나타날지는 이 데이터로 말할 수 없어요. 수치는 "이 조건에서 이렇게 관찰됐다" 정도로 읽어주면 돼요.

6종 중 일부(파생 상태 `useEffect`, 로직 중복 등)는 발현 횟수가 너무 적어서 해석을 보류했고요.

또 이 실험이 변수로 둔 건 'CLAUDE.md 지시' 하나뿐이에요. 작은 코드베이스에서 한 번씩만 생성했고, 에이전트가 내놓은 결과물을 품질 기준으로 다시 검사하는 단계는 넣지 않았어요. 그래서 팀 표준의 10.6%나 빈 catch의 절반 잔존을 '지시의 일반적인 한계'로 일반화하면 안 돼요 — 더 큰 코드베이스나 그런 검사 단계를 더하면 결과는 또 달라질 수 있으니까요.

바로 이 마지막 두 가지가 마음에 걸려서, 두 번째 실험을 했어요.

## 덧붙임 — 더 엄격한 두 번째 실험

위 한계에서 두 가지가 걸렸어요. 하나는 코드베이스가 만든 차이가 사실상 두 패턴(`any`·빈 catch)에 거의 다 기대고 있었다는 점이에요. 다른 하나는, 에이전트가 코드를 다 쓴 _뒤에_ 자기 결과물을 한 번 더 점검하게 하면 어떻게 되는지 안 해봤다는 점이고요.

그래서 같은 질문을 다시 던지되, 이번엔 **재는 방법을 바꿔서** 두 번째 실험을 했어요. 첫 실험처럼 설계는 결과를 보기 전에 미리 고정해 뒀고요. 바꾼 건 크게 세 가지예요.

**1) 재는 쪽을 바꿨다 — 린터 대신 AI 검토자**

첫 실험은 정적 분석 도구인 린터 6종으로 "정해진 나쁜 패턴이 새 코드에 몇 번 나왔나"를 셌어요. 기계 규칙이라 깔끔하지만, "문법은 맞는데 맥락상 나쁜" 코드는 못 잡아요. 그래서 이번엔 사람 대신 코드를 읽고 품질을 판정하는 LLM을 AI 검토자로 두고, 에이전트가 새로 쓴 코드를 프론트엔드 품질 기준 **28개 항목**에 비춰 **이 AI 검토자가 한 항목씩 "문제 있음 / 괜찮음 / 판단 보류"로 판정**하게 했어요. 기준은 네 영역으로 나뉘고, 영역마다 이런 항목들을 봤어요.

**상태 관리**

- 서버에서 받아온 데이터와 클라이언트 UI 상태를 구분하고, 로딩·에러를 손수 다시 만들지 않았는가
- 한 비동기 작업의 상태를 여러 boolean으로 쪼개지 않고 하나의 status로 다뤘는가
- 기존 상태·props에서 계산할 수 있는 값을 별도 상태로 중복 보관하지 않았는가
- 국소 UI 상태를 불필요하게 전역으로 끌어올리지 않았는가

**아키텍처**

- 데이터 페칭·비즈니스 로직을 렌더 컴포넌트와 분리했는가
- 모듈 의존 방향이 한쪽으로 흐르고, 책임이 한 레이어에 모여 있는가
- 새 로직을 가까운 모듈에 떠넘기지 않고 책임에 맞는 자리에 뒀는가

**컴포넌트 설계**

- 한 컴포넌트가 하나의 책임에 집중하는가
- 모순될 수 있는 값을 별개 boolean으로 쪼개지 않아, prop이 호출부에서 명료하게 읽히는가
- 반복되는 구조는 재사용 단위로 모으되, 한 번 쓰는 코드를 과하게 미리 추상화하지는 않았는가

**디자인 시스템**

- 색·간격·타이포 같은 시각 값을 토큰·상수 한곳에서 참조했는가
- 같은 의미의 값을 한 가지 표기로 통일했는가
- 같은 역할의 UI를 매번 새로 짜지 않고 재사용 단위로 모았는가

이 항목들을 더 잘게 나눈 게 28개이고, 핵심 비교엔 그중 23개를 썼어요.

**2) 더 많이, 더 안정적으로**

96번 실행했어요. 그리고 한 번 실행할 때마다 AI 검토자에게 판정을 **세 번 따로** 받았어요. 세 번 다 같은 결론이 나오는지 보려는 거예요. 결과적으로 세 판정이 84%에서 만장일치였어요 — 판정이 들쭉날쭉하지 않고 꽤 안정적이라는 뜻이죠.

**3) 이번에 새로 넣어 본 것 — 작성 후 '자기점검'**

첫 실험에서 바꿔 본 건 코드를 짜기 _전에_ 주는 지시 문서인 CLAUDE.md였어요. 이번엔 위에서 "안 해봤다"고 한 바로 그것, **코드를 다 쓴 *뒤에* 에이전트가 자기 코드를 스스로 다시 보고 한 번 고치게 하는 단계**를 넣어 봤고, 이 단계를 앞으로 '자기점검'이라고 부를게요. 이렇게 시켰고요.

> 구현을 끝낸 뒤, 제출하기 전에 네 가지 영역(상태 관리·아키텍처·컴포넌트 설계·디자인 시스템) 기준으로 코드를 한 번 다시 검토하고, 한 번 고쳐라.

여기서 한 가지가 중요해요. 점검 기준을 *"주변 코드와 똑같이 맞춰라"*가 아니라 *"이번에 새로 짠 코드 자체가 이 속성을 갖췄나"*로 적었어요. 주변 코드가 더러우면 "주변에 맞춰라"가 곧 "더럽게 짜라"가 되니까, "주변이 어떻든, 이 코드만 놓고 봤을 때 괜찮은가"를 묻도록 한 거죠.

> **잠깐 — 첫 실험의 '팀 표준'과 헷갈리지 마세요.** 첫 실험에서 효과를 본 방법(CLAUDE.md에 팀 표준을 적어 두는 것)은 코드를 짜기 _전에_ 미리 주는 지시였어요. 이번에 넣은 건 코드를 짠 _뒤에_ 스스로 점검하게 하는 단계라, 둘은 서로 다른 방법이에요. 그러니 "두 번째 실험이 첫 실험의 결과 ②를 강화했다"고 읽으면 안 돼요. 같은 걸 한 번 더 한 게 아니거든요.

### 결과 ① — 코드베이스 효과가 더 단단해졌다

"코드베이스에 나쁜 코드가 많으면 에이전트도 따라 한다"는 첫 실험의 발견이, 재는 방법을 통째로 바꾼 이번에도 똑같이 다시 나왔어요.

에이전트가 그 문제를 일으킬 만한 상황, 이를테면 색을 직접 다뤄야 하는 작업이었을 때 **AI 검토자가 실제로 "문제 있음"이라고 본 비율**은 이렇게 갈렸어요.

| 코드베이스                         | AI 검토자가 "문제 있음"으로 본 비율 |
| ---------------------------------- | ----------------------------------- |
| 더러운 코드베이스 (나쁜 코드 많음) | **83%**                             |
| 깨끗한 코드베이스 (깔끔함)         | 29%                                 |

54%p 차이예요. 방향도, 격차도 첫 실험과 같아요.

> **이 숫자를 첫 실험과 직접 비교하진 마세요.** 첫 실험의 6.1% / 22.7%와 여기 29% / 83%는 **재는 자가 아예 달라요.** 첫 실험은 린터가 센 비율이고, 이건 AI 검토자가 더 넓은 기준으로 본 비율이라 분모도 기준도 달라요. "22.7%가 83%로 늘었다"처럼 읽으면 틀려요. 가져갈 건 숫자 자체가 아니라, **방향과 큰 격차가, 더 엄격하게 재고 더 많이 반복해도 또 나왔다**는 사실이에요.

그리고 첫 실험이 걱정했던 것 — "효과가 두 패턴(`any`·빈 catch)에만 기댄 것 아니냐" — 도 풀렸어요. 따라 하기는 한 가지가 아니라 **세 가지 모양**으로 나타났고, 셋 다 더러운 코드베이스에서 훨씬 잦았거든요.

| 따라 하기의 모양                                 | 더러운 코드베이스 | 깨끗한 코드베이스 |
| ------------------------------------------------ | ----------------- | ----------------- |
| ① 같은 나쁜 패턴을 새 코드에 다시 씀             | 잦음              | 드묾              |
| ② 새 기능을 이미 비대한 파일에 그냥 얹음         | 24번 중 11번      | 0번               |
| ③ 시드에 있던 엉성한 컴포넌트를 그대로 가져다 씀 | 24번 중 17번      | 0번               |

②는 "새 코드를 깔끔한 새 자리에 두는 대신, 이미 너무 많은 일을 하는 파일에 또 얹는다"는 거예요. ③은 "옆에 있던 허술한 부품을 그대로 재사용한다"는 거고요. 둘 다 깨끗한 코드베이스에선 한 번도 안 일어났어요.

단, 깨끗한 쪽이 0인 건 *그쪽엔 비대한 파일도, 엉성한 부품도 애초에 없어서*인 면이 커요. 그러니 더 정확히 말하면 이거예요 — **나쁜 구조가 주어지면 절반에서 3분의 2의 런이 그걸 그대로 답습했고, 자기점검도 그건 못 막았다.**

**가장 선명한 예는 '디자인 토큰'이었어요.** 디자인 토큰은 색·간격 같은 시각 값을 코드에 직접 박지 않고, `--color-border` 같은 **이름**으로 한곳에서 관리하는 방식이에요. 나중에 색을 바꿀 일이 생겨도 이름이 가리키는 값 하나만 고치면 되니까요. 에이전트가 새 스타일을 쓸 때 이 이름을 참조하는지 봤어요.

| 코드베이스        | 색을 직접 박은(토큰을 안 쓴) 비율 |
| ----------------- | --------------------------------- |
| 더러운 코드베이스 | **21번 중 21번 (전부)**           |
| 깨끗한 코드베이스 | 21번 중 1번                       |

더러운 코드베이스에선 기회가 있던 **전부**가 색을 그대로 박았어요. 실제로 한 런은 새 스타일에 이렇게 썼고요.

```css
/* 더러운 코드베이스 — 색을 이름 없이 그대로 박음 */
border: 1px solid #d1d5db;
color: #6366f1;
```

같은 작업을 깨끗한 코드베이스에서 시킨 런은, 박힌 색 대신 이름 있는 토큰을 참조했어요. 코드베이스가 보여준 방식을 에이전트가 그대로 따라간 거예요.

### 결과 ② — 작성 후 자기점검은 '일부만' 막았다

그럼 이번에 넣은 자기점검 — 다 쓴 뒤 스스로 다시 보고 한 번 고치게 한 단계 — 은 효과가 있었을까요? 반은 맞고 반은 아니에요.

겉으로 드러나는 따라 쓰기는 확실히 줄었어요. 더러운 코드베이스의 "문제 있음" 비율이 **83%에서 51%로** 내려갔거든요. 스스로 한 번 더 보게 하니, 눈에 띄는 건 손봤다는 뜻이죠.

자기점검은 _전반적으로_ 품질을 올렸어요 — 깨끗한 코드도 29%→15%로 내려갔거든요. 그리고 그 위에, **더러운 코드 탓에 새로 생긴 결함을 한 겹 더 깎았어요**(83%→51%로, 깨끗한 쪽 감소폭의 두 배가 넘어요). 일반적인 청소 효과에 더해 *더러움 때문에 묻은 군더더기*까지 추가로 걷어낸 셈이죠.

그런데 **더러운 코드에 가장 진하게 박혀 있던 패턴**, 그래서 에이전트가 가장 자주 따라 쓰던 패턴은 자기점검으로도 거의 안 빠졌어요.

- '엉뚱한 곳에 얹기'와 '엉성한 부품 그대로 쓰기'는 별로 줄지 않았어요.
- 가장 굳은 디자인 토큰은 **꿈쩍도 안 했어요.** 색을 이름 없이 직접 박는 비율은 자기점검을 줘도 100%에서 100% 그대로였어요.

이건 첫 실험의 교훈과 정확히 같은 모양이에요. 첫 실험에서 "어떤 지시 문서도 모든 안티패턴을 똑같이 막진 못한다"를 봤는데, **방법을 바꿔도 그 교훈은 그대로**였어요. 작성 전 지시든 작성 후 점검이든, 표면적인 건 줄지만 코드베이스에 가장 깊이 박힌 패턴은 끝까지 남았어요. 여기서는 색을 이름 없이 직접 박는 것이 바로 그런 패턴이었고요.

### 두 번째 실험에서 가져갈 것

두 가지예요.

1. **"코드베이스가 에이전트 결과물을 바꾼다"는 발견이 더 단단해졌어요.** 이번엔 린터 6종이 아니라 AI 검토자가 더 넓은 품질 기준으로 봤고, 96번 반복했고, 설계를 미리 고정했는데도 같은 방향과 큰 격차가 또 나왔어요.
2. **어떤 한 가지 방법도 만능은 아니에요.** 작성 후 자기점검은 표면은 줄였지만, 코드베이스에 가장 깊이 박힌 패턴은 그대로 남겼어요.

물론 한계는 그대로예요. 이 실험도 여전히 한 가지 기술 묶음(Vite + React + TypeScript)의 같은 계열 코드베이스 위에서 한 거라, 다른 언어나 더 큰 프로젝트에서도 같을지는 이 데이터로 말할 수 없어요.

이번 방식 특유의 단서도 둘 있어요. 첫째, 품질을 판정한 게 사람이 아니라 **AI 검토자**예요. 채점한 모델과 코드를 짠 모델이 같은 계열이라, 둘이 같은 약점을 공유하면 그 부분은 못 걸러요. 둘째, 어떤 항목이 더러운 쪽에서 0%로 나왔다고 해서 꼭 '에이전트가 잘 버텼다'는 뜻은 아니에요. **이번 태스크들이 그 항목을 건드릴 일 자체가 없었던** 경우도 섞여 있거든요. 그러니 수치는 "이 조건에서 이렇게 관찰됐다" 정도로 읽어주세요.

그래서 이 실험이 _정직하게_ 말할 수 있는 건 여기까지예요 — **나쁜 코드베이스는 에이전트를 물들이고, 말로 된 지시나 스스로의 점검만으로는 가장 깊이 밴 패턴까지 막진 못한다.** 그렇다면 어기면 _막아 버리는_ 결정적 장치 — 차단 게이트나 독립된 검증 단계 — 는 이 고리를 끊을 수 있을까? 그건 이 실험이 답하지 못한, 다음 질문이에요.
